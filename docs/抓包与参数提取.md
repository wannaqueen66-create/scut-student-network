# 抓包与参数提取

本文档介绍如何在**合法授权**的前提下，抓取校园网 Portal 登录请求，并将其中的关键参数提取到本项目配置文件中。

> 说明：不同学校、校区、运营商和 Portal 厂商的接口都可能不同。本文以常见 ePortal/Portal 登录流程为例，实际字段名请以你自己的抓包结果为准。

## 准备工作

在开始前，请准备：

- 一台已经连接到校园网、且会跳转到 Portal 登录页的设备
- Chrome 或 Edge 浏览器
- 本项目中的 `portal_login.conf.example`

建议在抓包前：

- 关闭浏览器中与校园网无关的标签页，减少干扰请求
- 使用无痕模式测试一次，避免历史缓存干扰
- 如果已经在线，先退出登录，再重新登录

## 方法一：通过浏览器开发者工具抓包

这是最推荐的方法。

### 步骤 1：打开 Portal 登录页

先打开你学校或单位的 Portal 登录页。

例如，部分 ePortal 页面可能长这样：

- `https://example.edu.cn/a79.htm?wlanacname=example`

> 注意：这只是示例。请以你自己的登录页地址为准。

### 步骤 2：打开开发者工具

- 按 `F12`
- 或右键页面，选择“检查”

### 步骤 3：切到 Network 面板

在开发者工具顶部选择：

- `Network` / `网络`

如果没看到：

- 点击顶部的 `>>`，再选择 `Network`

### 步骤 4：设置抓包选项

建议启用以下选项：

- `Preserve log` / `保留日志`
- `Disable cache` / `禁用缓存`（建议开启）

然后：

- 点击左上角清空按钮，先把旧请求清掉
- 确保左上角录制按钮是红色，表示正在记录

### 步骤 5：重新执行一次登录

为了抓到真正的登录请求，建议按这个顺序操作：

1. 如果当前已经在线，先点击页面上的“注销 / Logout”
2. 等待回到登录页
3. 输入账号密码
4. 点击“登录 / Login”

### 步骤 6：定位登录请求

在 Network 请求列表中，重点找以下几类请求：

- 包含 `login` 的请求
- 包含 `ACSetting` 的请求
- 包含 `portal`、`auth`、`signin`、`callback` 的请求

常见形式包括：

- `/eportal/?c=ACSetting&a=Login...`
- `/eportal/portal/login?...`
- `/login?callback=...`

你可以在顶部过滤框中依次尝试输入：

- `login`
- `ACSetting`
- `portal`

### 步骤 7：查看请求详情

点击疑似登录请求后，在右侧重点查看：

- `Headers` / `标头`
- `Payload` / `负载`
- `Response` / `响应`

优先记录以下信息：

- `Request URL`
- `Request Method`（GET 或 POST）
- `Query String Parameters` 或 `Form Data`
- `Request Headers` 中的重要字段（如 `Referer`、`User-Agent`）
- `Response` 中的返回内容（例如 `result`、`msg`、`ret_code`）

### 步骤 8：复制为 cURL（推荐）

如果浏览器支持：

- 右键该请求
- 选择 `Copy` / `复制`
- 再选 `Copy as cURL (bash)` / `复制为 cURL (bash)`

这样最方便后续提取参数。

> 安全提示：如果你打算把 cURL 发给别人协助分析，请先手动打码账号、密码、Cookie 等敏感信息。

## 方法二：查看表单提交地址

如果在 Network 里始终找不到明显的登录请求，可以尝试查看页面源码：

1. 在登录页按 `Ctrl+U`
2. 搜索：
   - `form`
   - `action=`
   - `ACSetting`
   - `login`
3. 找到表单提交地址和输入框字段名

你需要重点关注：

- 表单的 `action`
- 用户名输入框的 `name`
- 密码输入框的 `name`
- 是否有隐藏字段（如 `wlanacname`、`user_ip`、`mac`、`callback`）

这种方法适合传统表单提交流，但如果页面是 JavaScript 动态发请求，还是建议回到 Network 面板抓包。

## 你需要提取哪些参数

不同 Portal 不完全一样，但通常要整理出以下内容：

### 1. 登录地址

例如：

- `https://your-portal-host:port/eportal/portal/login`

对应配置：

- `HOST`
- `PORT`
- 可选的 `HOST_IP`

获取方法：

- `HOST`：查看登录请求的 `Request URL`，域名部分就是 `HOST`
- `PORT`：查看登录请求的 `Request URL`，端口部分就是 `PORT`
- 如果 URL 中未显式写端口，通常：
  - `http` 对应 `80`
  - `https` 对应 `443`
- `HOST_IP`：不是必须项。仅当你测试发现 DNS 不稳定，或者脚本环境下需要固定解析时，再通过以下命令获取：
  - `nslookup your-portal-host`
  - `ping your-portal-host`
  - `getent hosts your-portal-host`

### 2. 接入点 / AC 名称

例如：

- `wlan_ac_name`
- `wlanacname`

对应配置：

- `AC_NAME`

获取方法：

- 先看登录页 URL，常见情况会出现 `wlanacname=xxx`
- 再看登录请求参数，常见字段为 `wlan_ac_name=xxx`
- 最终以实际登录请求里的值为准

### 3. 用户名和密码字段名

常见字段：

- `user_account`
- `user_password`

本项目默认配置项：

- `PORTAL_USER`
- `PORTAL_PASS`

> 注意：项目中的配置变量名可以和接口字段名不同，脚本里会负责映射。

### 4. 本机 IP / MAC 相关字段

常见字段：

- `wlan_user_ip`
- `wlan_user_mac`
- `wlan_user_ipv6`

本项目脚本默认会自动读取：

- 网卡 IPv4
- 网卡 MAC 地址

### 5. 其他固定参数

有些 Portal 会要求额外参数，例如：

- `callback`
- `login_method`
- `jsVersion`
- `terminal_type`
- `lang`
- `mac_type`
- `program_index`
- `page_index`
- `v`

这些参数里有些是固定值，有些是时间戳或随机字符串。需要以抓包结果为准。

## 如何填写到配置文件

1. 复制示例配置：

```sh
cp portal_login.conf.example portal_login.conf
```

2. 按抓包结果修改配置，例如：

```sh
PORTAL_USER="your-id"
PORTAL_PASS="your-password"
IFACE="apclix0"
AC_NAME="example_ac"
HOST="portal.example.edu.cn"
HOST_IP=""
PORT="802"
```

3. 如果你的学校接口字段和本项目默认脚本不同，需要进一步修改脚本中的请求参数。

## 常见登录流类型

### 类型 A：传统表单流

典型特征：

- 请求路径类似 `ACSetting&a=Login`
- 页面存在明显的 `<form action="...">`
- 浏览器点击登录后直接发起表单请求

### 类型 B：AJAX / JSONP 登录流

典型特征：

- 请求路径类似 `login?callback=...`
- 页面由 JavaScript 发起请求
- 返回内容可能是 JSON、JSONP 或文本片段

### 类型 C：重定向认证流

典型特征：

- 点击登录后跳转多个地址
- 中间涉及统一认证、CAS、运营商认证页
- 可能需要 Cookie、Token、动态签名

如果你的网络属于类型 C，本项目模板可能需要较大改动。

## 常见问题

### 1. 抓不到登录请求怎么办？

建议检查：

- 是否已经勾选 `Preserve log`
- 是否清空了旧请求
- 是否确实重新执行了一次“注销 → 登录”流程
- 是否在过滤框里输入了过窄的关键词

可以先清空过滤词，再重新观察所有请求。

### 2. 找到了请求，但不知道哪几个参数必须保留？

先全部记录下来，再逐步区分：

- 账号密码类参数
- 本机信息类参数（IP/MAC）
- 固定参数
- 动态参数（时间戳、随机值）

实在不确定时，优先保留抓包中出现的全部参数，再慢慢做减法。

### 3. 密码能不能直接写进仓库？

不能。

请始终：

- 使用本地配置文件保存账号密码
- 不要提交真实配置到 Git 仓库
- 不要把完整 cURL、Cookie、Token 原样公开到 issue 或 README

### 4. 为什么脚本里有 `HOST_IP`？

有些 Portal 环境下，域名解析不稳定，或脚本运行环境 DNS 行为和浏览器不同，因此项目模板保留了 `--resolve` 这一类能力。实际是否需要，请以你的环境测试结果为准。

### 5. 为什么脚本不能保证适配所有学校？

因为不同网络环境差异很大，包括：

- 接口路径不同
- 参数名不同
- 是否依赖 Cookie/Token 不同
- 是否需要 JavaScript 生成签名不同

本项目更适合作为**分析与自动化模板**，不是对所有校园网都直接可用的通杀脚本。
